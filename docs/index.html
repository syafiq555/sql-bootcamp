Advanced Filtering: IN and LIKE

IN: Cleaner than multiple OR statements

-- Instead of multiple OR:
WHERE city = 'Seattle' OR city = 'Austin' OR city = 'Denver';

-- Use IN (cleaner):
WHERE city IN ('Seattle', 'Austin', 'Denver');

LIKE: Pattern matching for strings

WHERE customer_name LIKE 'A%';     -- Starts with 'A'
WHERE customer_name LIKE '%son';   -- Ends with 'son'
WHERE email LIKE '%@gmail.com';    -- Contains '@gmail.com'
WHERE code LIKE 'A_C';             -- A, any character, C
IN is much more readable than chaining multiple OR conditions, especially with many values. LIKE uses wildcards: % (any number of characters), _ (exactly one character). Common mistake: using = with LIKE patterns won't work. Must use LIKE operator.

The Concept of NULL (Unknown Data)

NULL is NOT zero, NOT empty string, NOT false.

NULL = "unknown" or "missing" data

NULL Unknown/missing
0 A value (the number zero)
'' A value (empty string)

⚠️ Gotcha: Cannot filter NULL with = or !=

-- ❌ WRONG
WHERE phone_number = NULL;

-- ✅ CORRECT
WHERE phone_number IS NULL;
WHERE phone_number IS NOT NULL;
NULL is a notoriously confusing concept. This is one of the most common mistakes. Students will try WHERE column = NULL and get no results, which confuses them. Aggregates handle NULL specially: COUNT(*) counts all rows including NULLs, but COUNT(column) ignores NULLs. LEFT JOINs introduce NULLs when there's no match - we'll see this next in the JOINs section.

How Aggregates Handle NULL

COUNT(*) Counts ALL rows (including NULLs)
COUNT(column) Counts non-NULL values only
SUM(), AVG(), etc. Ignore NULL values

Example: 5 employees, 2 with NULL phone numbers

SELECT COUNT(*) as all_rows,
       COUNT(phone_number) as with_phones
FROM employees;

-- Result: all_rows = 5, with_phones = 3
This is a critical distinction. COUNT(*) is useful for "how many total records", but COUNT(column) is useful for "how many records have this data". This causes real bugs: students might count records and find 5 total, but only 3 have phone numbers. The difference is the 2 NULLs.

Aggregating Data

COUNT() Count rows
SUM() Total of numeric column
AVG() Average value
MIN() / MAX() Minimum / Maximum

Example: Total sales and average order value

SELECT COUNT(*) as total_orders,
       SUM(amount) as total_sales,
       AVG(amount) as avg_order
FROM orders;

Using Aliases for Readability

Column Aliases: Rename columns in the result set

SELECT AVG(amount) AS average_sale,
       MAX(amount) AS highest_sale,
       COUNT(*) AS total_orders
FROM orders;

Table Aliases: Shorten table references (required for SELF JOINs)

-- Instead of:
SELECT customers.name, orders.amount
FROM customers INNER JOIN orders ON customers.customer_id = orders.customer_id;

-- Use table aliases (cleaner):
SELECT c.name, o.amount
FROM customers c INNER JOIN orders o ON c.customer_id = o.customer_id;
Column aliases (AS) are used to rename columns in the output. This is useful for aggregates: AVG(amount) becomes readable as "average_sale". Table aliases are abbreviations for table names. Using 'c' for customers and 'o' for orders makes JOIN queries much shorter. Aliases are REQUIRED for SELF JOINs (joining a table to itself), since you need to reference the same table twice with different names.

Filtering Groups with HAVING

Key difference:

🔍 WHERE filters rows BEFORE grouping

🔍 HAVING filters groups AFTER grouping

SELECT region, SUM(amount) as total_sales
FROM orders
GROUP BY region
HAVING SUM(amount) > 6000;

Result: Only regions with sales over $6,000 are shown (West: $7,320.50)

HAVING is the "GROUP BY WHERE" clause. Use it when you need to filter based on aggregated values. Example: "Show only regions with total sales > $6,000" - you must use HAVING because you're filtering on SUM(amount), which is an aggregate. If you tried WHERE SUM(amount) > 6000, it would fail - WHERE can't use aggregates. Common mistake: Students use WHERE when they should use HAVING on aggregate conditions.

Database Design: Why Normalize?

Problem: Storing data poorly causes redundancy and errors

customers table (BAD):
customer_id | name  | order1  | order2  | order3
1           | Alice | 100     | 200     | NULL

Solution: Split into separate tables with keys

customers: customer_id, name
orders: order_id, customer_id, amount
## The Problem with Bad Design **Unnormalized table (Denormalized):** ``` customer_id | name | order1 | order2 | order3 1 | Alice | 100 | 200 | NULL ``` **Problems:** - Alice's name is stored 3 times (what if she moves?) - If we add more orders, we need more columns (doesn't scale!) - What if order1 doesn't exist? Column wastes space ## The Solution: Separate Tables **Customers table:** ``` customer_id | name 1 | Alice ``` **Orders table:** ``` order_id | customer_id | amount 1 | 1 | 100 2 | 1 | 200 3 | 1 | 300 ``` **Benefits:** - ✅ Alice's name stored once (easy updates) - ✅ Can add unlimited orders (just add rows) - ✅ No wasted space (no NULL columns) ## The Rule **Normalize:** One fact per place. If you need to update it, update it once, everywhere updates automatically (via foreign key relationships).

Normal Forms (Simplified)

1NF All columns have single values (no repeating groups)
2NF No partial dependencies (all non-key columns depend on entire key)
3NF No transitive dependencies (non-key columns don't depend on other non-key columns)

Practical tip: Most production databases follow 3NF. It's the sweet spot between structure and query simplicity.

## Don't Memorize, Understand the Principle ### 1NF - First Normal Form **Rule:** Each cell has only one value (no lists/arrays) **Bad:** `courses: "Math, Physics, Chemistry"` **Good:** Separate rows for each course ### 2NF - Second Normal Form **Rule:** Non-key columns must depend on the ENTIRE key (not part of it) **Bad:** Table with key `(student_id, course_id)` but has a column `student_name` (depends only on student_id, not whole key) **Good:** Move `student_name` to separate `students` table ### 3NF - Third Normal Form **Rule:** Non-key columns depend only on the key, not on each other **Bad:** `employees` table with columns: `employee_id`, `department_id`, `department_name` (department_name depends on department_id, not on employee_id) **Good:** Move `department_name` to separate `departments` table ## In Practice **Most production databases are 3NF** because: - ✅ Eliminates redundancy (saves storage) - ✅ Makes updates safe (change once, applies everywhere) - ✅ Maintains data integrity (no contradictions) - ✅ Queries are still simple with JOINs **Take-home:** If you see duplicated data, that's a sign the schema needs normalization.

Advanced JOINs

FULL OUTER JOIN All rows from both tables (with NULLs where no match)
CROSS JOIN Cartesian product - all combinations
SELF JOIN Join a table to itself (e.g., employee-manager)

## Quick Recap of All JOINs ### INNER JOIN (Session 2 review) Returns only matching rows from both tables ```sql SELECT * FROM customers INNER JOIN orders ON customers.customer_id = orders.customer_id; ``` ### LEFT JOIN (Session 2 review) Keep ALL from left table + matching from right (NULLs if no match) ```sql SELECT * FROM customers LEFT JOIN orders ON customers.customer_id = orders.customer_id; -- Shows all customers, even those with no orders ``` ### RIGHT JOIN Opposite of LEFT - keep all from right table ### FULL OUTER JOIN (Advanced) Keep ALL from both tables (NULLs where no match) ```sql SELECT * FROM customers FULL OUTER JOIN orders ON customers.customer_id = orders.customer_id; -- Shows customers with/without orders AND orders with/without customers ``` ### SELF JOIN (Advanced) Join a table to itself (hierarchical data like manager-employee) ```sql SELECT e.name as employee, m.name as manager FROM employees e LEFT JOIN employees m ON e.manager_id = m.employee_id; -- e = employee, m = manager (same table!) ``` ### CROSS JOIN (Advanced) Cartesian product - ALL combinations ```sql SELECT * FROM colors CROSS JOIN sizes; -- If colors has 5 rows, sizes has 3 rows → result has 15 rows ``` ## When to Use Each | JOIN | Use when... | |---|---| | INNER | You only want matching records | | LEFT | You want all from left, even if no match | | FULL OUTER | You want to find unmatched records in either table | | SELF | Hierarchical data (boss/employee, parent/child) | | CROSS | Generate combinations (unlikely in practice) |

Indexes: Speed Up Queries

Analogy: Index in a book vs scanning every page

Indexes are sorted lookups on columns

-- Create an index on frequently queried columns
CREATE INDEX idx_customer_id ON orders(customer_id);
CREATE INDEX idx_order_date ON orders(order_date);

-- Now queries using these columns are much faster!
SELECT * FROM orders WHERE customer_id = 5;

Trade-off: Indexes speed up SELECT but slow down INSERT/UPDATE (more to maintain)

## Why Indexes Matter **Without index:** Database scans *every single row* (Table Scan) = SLOW ❌ **With index:** Database jumps directly to matching rows (Seek) = FAST ✅ ### Book Analogy - No index = reading every page to find a topic - Index = jump to page numbers in the back, then go directly to relevant pages ## When to Create Indexes Index columns that are frequently: - In WHERE clauses (`WHERE customer_id = 5`) - In JOIN conditions (`ON orders.customer_id = customers.customer_id`) - In ORDER BY or GROUP BY ## ⚠️ The Trade-Off **Pros:** SELECT queries become MUCH faster **Cons:** INSERT/UPDATE/DELETE become slower because the database must update the index too. Like updating a book's index every time you add a new page. ## Rule of Thumb - Read-heavy tables (reporting, analytics) → many indexes ✅ - Write-heavy tables (real-time data) → fewer indexes ⚠️

DESC:** Sort products by revenue (highest first) - **ROW_NUMBER():** Assigns rank 1, 2, 3, 4, 5... to each product - **Key difference from GROUP BY:** Each product row stays intact (we don't lose data) The CTE now has 10 rows with revenue + rank columns. ### Step 3: Filter Top 5 ```sql SELECT * FROM product_revenue WHERE rank <= 5 ``` - Takes the 10 ranked products - Returns only rows where rank is 1-5 - **Result:** Top 5 revenue-generating products ## Why Use a CTE Here? **Without CTE (messy):** ```sql SELECT * FROM ( SELECT p.product_id, SUM(...), ROW_NUMBER() OVER (...) FROM products p JOIN order_details od ... GROUP BY ... ) WHERE rank <= 5 ``` **With CTE (clean):** - Named intermediate result (`product_revenue`) makes intent clear - Can reference the CTE multiple times if needed - Easier to debug (can run `SELECT * FROM product_revenue` alone) ## Teaching Approach 1. **Start simple:** `SELECT * FROM products` (show 10 products) 2. **Add JOIN:** `INNER JOIN order_details` (show 30 rows) 3. **Add aggregation:** `GROUP BY + SUM()` (collapse to 10 rows with revenue) 4. **Add window function:** `ROW_NUMBER()` (add rank column, still 10 rows) 5. **Filter:** `WHERE rank <= 5` (final 5 rows) **Key insight:** Build incrementally, verify each step before adding complexity! ## Real-World Impact This query type is used for: - **E-commerce dashboards:** "Top 10 products this month" - **Sales analytics:** "Which products drive revenue?" - **Inventory decisions:** "Stock more of products ranked 1-5" - **Marketing focus:** "Promote top sellers" Students should understand: This isn't academic - it's business-critical analysis.

-- Month-over-month retention
WITH monthly_customers AS (
    SELECT
        YEAR(order_date) as year,
        MONTH(order_date) as month,
        COUNT(DISTINCT customer_id) as customers
    FROM orders
    GROUP BY YEAR(order_date), MONTH(order_date)
)
SELECT
    month,
    LAG(customers) OVER (ORDER BY year, month) as prev_month,
    customers as curr_month,
    ROUND(100.0 * customers / LAG(customers) OVER (...), 2) as retention_pct
FROM monthly_customers;

🧠 Critical Thinking Challenge

This query shows month-over-month growth in total active customers. But is this the same as retention?

Key question: If February has 450 customers and January had 500, does this prove that 90% of January's customers came back?

Hint: Those 450 February customers could be 450 brand-new people. You'd never know just by comparing totals!

💡 To find TRUE retention: Track the same customer_id across months. Find customers in both January AND February.

## What Is Retention Rate? **Retention = % of customers who returned to purchase again** from one month to the next. ### Simple Example - Jan: 500 customers made a purchase - Feb: 450 of those customers came back - **Retention = 450/500 = 90%** - High retention (80%+) = Loyal customers ✅ - Low retention (30-40%) = One-time buyers ❌ ## The LAG Function `LAG(customers)` = "look at the previous row's value" In the query: - Each month shows: previous month's count → current month's count - Then calculates: (current / previous) × 100 = retention % ## ⚠️ Important: This Isn't True Retention! This query has a **critical flaw** - it doesn't track *the same customers* across months. **What it actually does:** Compares total customer counts **Problem:** 450 customers in Feb could be 450 *different people entirely* **True retention needs:** Customer-level tracking (customer_id across months) ## Teaching Tip Ask students: "If Jan had 500 customers and Feb has 450, did 90% come back? How would you prove it?" → They'll realize you need to see the same customer_id in both months!

## How LAG() "Looks Back" to Previous Rows **LAG() is a window function** that accesses data from a previous row without using a self-join. ### Step-by-Step Execution **Step 1: CTE creates monthly summary** ```sql WITH monthly_customers AS ( SELECT YEAR(order_date) as year, MONTH(order_date) as month, COUNT(DISTINCT customer_id) as customers FROM orders GROUP BY YEAR(order_date), MONTH(order_date) ) ``` Result (before LAG): - Jan: 4 customers - Feb: 3 customers - Mar: 3 customers - Apr: 2 customers - May: 3 customers **Step 2: LAG() adds previous month's value to each row** ```sql LAG(customers) OVER (ORDER BY year, month) as prev_month ``` - **ORDER BY year, month:** Ensures months are in chronological order - **LAG(customers):** For each row, look at the previous row's "customers" value - **Result:** Each month now shows its own count + previous month's count ### How Retention % is Calculated ```sql ROUND(100.0 * customers / LAG(customers) OVER (...), 2) as retention_pct ``` **February example:** - Current month (Feb): 3 customers - Previous month (Jan): 4 customers - Calculation: (3 / 4) × 100 = 75% **March example (highlighted, perfect retention):** - Current month (Mar): 3 customers - Previous month (Feb): 3 customers - Calculation: (3 / 3) × 100 = 100% **May example (growth!):** - Current month (May): 3 customers - Previous month (Apr): 2 customers - Calculation: (3 / 2) × 100 = 150% (growth, not technically "retention") ### Why LAG() is Powerful **Without LAG (painful self-join):** ```sql SELECT m1.month, m2.customers as prev_month, m1.customers as curr_month FROM monthly_customers m1 LEFT JOIN monthly_customers m2 ON m2.month = m1.month - 1 AND m2.year = m1.year ``` **With LAG (clean):** ```sql SELECT month, LAG(customers) OVER (ORDER BY year, month) as prev_month ``` LAG eliminates complex self-joins for time-series comparisons! ### Critical Teaching Point: This Isn't True Retention **What this query ACTUALLY measures:** Total customer count comparison month-over-month **Example showing the flaw:** - Jan: Customers 101, 102, 103, 104 (4 customers) - Feb: Customers 105, 106, 107 (3 customers) - Query shows: 75% "retention" (3/4) - **Reality:** 0% retention! All February customers are NEW (none from January returned) **True retention would require:** ```sql -- Count customers who appear in BOTH months SELECT COUNT(DISTINCT customer_id) FROM orders WHERE MONTH(order_date) = 2 AND customer_id IN ( SELECT customer_id FROM orders WHERE MONTH(order_date) = 1 ) ``` ### Real-World Use Cases for LAG() ✅ **Month-over-month revenue comparison** ✅ **Stock price changes** (today's price vs yesterday's) ✅ **Website traffic trends** (this week vs last week) ✅ **Inventory levels** (current vs previous period) All these use LAG() to compare current row to previous row efficiently. ### Teaching Strategy 1. **Show the data flow:** CTE → LAG() adds prev_month column → retention calculation 2. **Highlight March row (100%):** "Same customer count = perfect retention" (or is it?) 3. **Ask critical question:** "Did the same 3 customers return in March? How would you verify?" 4. **Lead to realization:** Need to track customer_id across months (more advanced query) This builds analytical thinking - students learn to question metrics, not just calculate them!
